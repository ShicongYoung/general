# 协同任务增强项目 — 财年总结（自定义字段 × 工单发起）

**项目性质**：同一产品主线下的能力组合 — 在协同任务基础能力之上，补齐「可配置信息结构」与「从工单流入协同」两条关键链路。  
**适用场景**：财年复盘、向管理层/业务侧汇报进展与效果，**非**技术开发文档。

**效果快照时间**：下表核心指标统一取自 **自然周 2026-03-23（周一）～2026-03-29（周日）**。该周位于公司 **财年末期（3 月）**，与日常经营周报同源，便于与「财年内是否已经把能力用起来」对齐；若需全财年累计口径，可在 BI 中按财年整段（上年 4 月～本年 3 月）另做趋势与累计表。

---

## 一、为什么要做这件事（背景）

| 维度 | 说明 |
|------|------|
| **用户侧** | 工厂之间、任务类型之间差异大，固定表单难以覆盖「现场要记什么」；一线更希望按本厂、本任务类型灵活扩展字段，而不是被产品默认字段绑死。 |
| **流程侧** | 大量问题从**工单/现场执行**暴露，若必须跳出工单再起协同，路径长、容易断档；需要把「发现问题的现场」和「组织协同闭环」连起来。 |
| **经营侧** | 协同任务模块（基座）已单独跑通一段时间后，下一阶增长来自 **更深使用**（字段沉淀过程数据）和 **更广入口**（工单转化），属于同一阶段的自然延伸。 |

---

## 二、我们做了什么（解决方案与里程碑）

| 时间 | 交付 | 对用户的承诺 |
|------|------|----------------|
| **2025-06-26** | **协同任务模块**正式上线 | 支持「发起 → 响应 → 完成」主流程，形成可观测的使用基座。 |
| **2026-01-08** | **支持自定义字段** | 协同任务、任务处理记录可配置扩展字段，满足不同任务类型与工厂的填报诉求。 |
| **2026-01-08** | **支持从工单发起协同任务** | 在工单侧提供快速入口，新建协同与工单建立明确关联，强化生产场景连贯性。 |

**项目边界**：自定义字段与工单发起为 **同一阶段目标下的组合交付**，适合在财年总结中 **合并叙述**：既讲「能力是否被用起来」，也讲「两条能力是否相互放大」。

## 三、上线效果（量化，财年末代表周）

以下数据聚焦 **单周切片**，用于回答：**临近财年收尾，深度使用协同的规模、粘性，以及新能力渗透到了多少深度活跃用户**。

| 指标 | 数值 | 业务含义（一句话） |
|------|------|---------------------|
| **周活跃工厂**（一周内 ≥2 个自然日有创建协同） | **106** | 真正有「持续在用」习惯的工厂规模，是本项目的核心分母之一。 |
| **周用户留存**（上周同样「≥2 日活跃」的工厂中，本周仍达标比例） | **63%** | 习惯是否可延续；财年复盘里可配合趋势判断 「稳不稳」。 |
| **周内有创建行为的工厂**（≥1 日有创建） | **255** | 触达面；与上一行一起看，可知「浅尝」与「坚持用」的漏斗。 |
| **活跃转化**（周活跃 / 周内有创建） | **≈41.6%** | 从「用过」到「这周用得够勤」的转化效率。 |
| **自定义字段使用工厂**（当周产生有效取值，**不含空数组占位**） | **110** | 新能力已覆盖的工厂广度；统计时已排除无业务含义的空数据写法。 |
| **自定义字段在深度活跃用户中的渗透** | **约 37%**（**39 / 106**，内部复核） | 深度活跃工厂里，有多少家当周真的写入了有效自定义字段。对外材料曾表述为 **约 32%（34 / 106）**，建议在年度定稿时 **统一周次与去重规则** 后二选一对外。 |
| **从工单发起占比**（关联工单的协同任务 / 当周新建任务总数） | **≈16%**（当周快照） | 工单入口是否形成稳定习惯；与对外常说的「约两成」需 **对齐是否同一周、是否四舍五入**。 |

### 效果解读（结论先行）

- **协同基本盘**：财年末代表周上，**超百家工厂**保持「一周多日」的协同节奏，**六成以上**能在周维度维持连续活跃，说明模块已从「能上架」进入「能扎根」的阶段。  
- **增强项渗透**：自定义字段在 **百级工厂** 有有效写入；在 **周活跃** 工厂中的渗透约 **三分之一到四成**（视采用 34 还是 39 的统计口径），说明能力已被核心用户使用，但仍有空间通过 **模板、行业方案、培训** 拉高渗透。  
- **工单联动**：工单发起占比在当周快照约 **16%**，证明入口有价值，但尚未成为主路径；财年下一阶可重点看做 **场景包**（哪些工单类型优先露按钮）与 **成功路径打法**。

## 四、经验沉淀（产品向）

| 主题 | 沉淀 |
|------|------|
| **「一期基座 + 二期增强」叙事** | 先验证主流程与活跃、留存，再叠自定义与入口，避免尚未养成习惯就堆配置，降低支持成本与用户认知负担。 |
| **指标对外要「一套故事」** | 活跃定义、是否含删除数据、自定义是否排除空值，会直接影响渗透分子分母；**财年对外数字建议与数据/运营对齐一封口径说明**，避免汇报与宣传页打架。 |
| **组合发布要共用一个时间窗** | 同一迭代内多能力，复盘时应用 **同一周、同一活跃定义** 看交叉渗透，否则难以回答「是不是同一拨人在用」。 |
| **空数据与真实使用** | 自定义类能力易出现「有记录但无实质内容」；产品侧在 **埋点/报表** 上应尽早约定「何谓使用」，减少「虚高」带来的误判。 |

---

## 五、财年后续可行动方向（供规划引用）

1. **渗透率**：以「周活跃工厂」为锚，拆解自定义字段渗透的行业/规模分层，识别高潜客群与阻断原因（不会配、不敢配、不配也行）。  
2. **工单入口**：拉齐产品与数据对「工单发起占比」的周定义；针对占比偏低场景，用 **工单类型 + 工序** 做定向曝光而不是全流水线上按钮。  
3. **留存与深度使用**：在留存稳定的前提下，通过 **任务类型模板、默认字段包** 降低配置成本，把 37% 级别的渗透往更高台阶推。
